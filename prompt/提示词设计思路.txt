# 任务
要做什么；为什么做 / 结果给谁用；
做到什么程度算成功（验收标准：测试用例、性能指标、预算上限等）。

# 背景
与任务直接相关的事实、数据、现状、已有决策和假设。
大块材料（代码 / 文档 / 数据）用 XML 标签或分隔符包裹，与指令物理分开。

# 约束
硬性边界（技术、预算、时间、兼容性……）及冲突时的优先级。
异常处理规则：如果 X，则 Y；信息不足时明确说明并提问，不要编造。

# 输出
格式、结构、语言、篇幅、受众。
需要人工复核时：要求结论附上关键依据与假设（而非复述完整推理过程）。


原则：七条
定义问题，不规定解法。 目标、目的、验收标准写清楚，解题路径留给模型。想控制的只是思考投入量（一句话或 API 参数），不是思考步骤。

指令做减法，事实做足量。 删除教步骤的指令、增智型人设、情绪激励和重复强调；补齐决策所需的背景、数据和假设。判据是相关性，不是长度。

约束与验收标准是最大的杠杆。 边界越明确，解越对口；具体的成功标准会被模型用于内部自检，比"请认真检查"有效得多。禁止性规则照常写，无需迂回正述。

结构分隔，措辞自洽。 用标题或标签把指令、材料、约束隔开；发布前自查约束之间是否矛盾——推理模型会死抠字眼，矛盾指令会让它把思考预算浪费在猜你的意图上。

示例只定格式，不教推理。 零样本优先；必须给示例时只给输入-输出对，注明"仅为格式示例，勿套用其中逻辑"。

角色只定视角与受众，不用于增智。 "面向 CTO 的技术决策报告""从安全审计视角审查"有用；"你是智商200的资深专家"没用。

一句话收束：把提示词当成写给一位资深工程师的需求单——已知条件写准，验收标准写死，禁区画清，交付格式说明白，然后让他自己干活。这套写法对推理模型最优，对非推理模型也不吃亏，因此是当下最简洁且最不易失效的默认方案。
