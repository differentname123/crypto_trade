### 一、 技术选型方案

本系统采用基于云端边缘计算的全栈无服务器（Serverless）架构，将前端页面、后端逻辑与数据存储完全托管于 Cloudflare 生态中。

| 模块组件 | 选型技术 | 选型原因与客观技术指标 |
| --- | --- | --- |
| **计算与分发平台** | **Cloudflare Workers**<br>

<br>(Runtime: V8 Engine) | 具备全球分布式低延迟响应能力。免去传统服务器的运维成本。单次请求最大内存限制为 128 MB（需在代码层面严格控制数据加载量）。 |
| **数据存储方案** | **Cloudflare D1**<br>

<br>(Serverless SQLite) | 关系型数据库。**免费版额度：每天 100,000 次读取操作，100,000 次写入操作。** 能够支持高频的日志追加与 K 线大文本覆写，远超 KV 存储的写入限额。 |
| **系统架构形态** | **单文件架构 (Single-file Monolith)** | 前端静态页面（HTML/CSS/JS）作为字符串常驻于 Worker 后端代码中。通过路由分发（Routing）在单一文件内完成网页渲染与 API 接口响应，最小化部署复杂度。 |
| **数学计算引擎** | **原生 JavaScript (ES6+) 标准内置对象** | 舍弃 Python 体系的 Pandas、Numpy 和 CCXT 库。利用原生 Array 对象的 `map`、`reduce`、`filter` 以及 `Math` 标准库手搓重采样和指标公式，以在 128MB 内存限制内提供微秒级的计算性能。 |

### 二、 数据库设计 (Cloudflare D1 Schema)

在 D1 实例中创建两张独立的数据表，分别承载“指标计算基础数据”与“策略推演结果流水”。

#### 1. 基础 K 线缓存表 (`kline_cache`)

用于缓存 1m 粒度的基础 K 线，避免每次重复拉取历史数据，降低触发表风控的概率。由于 D1 不适合单条存储百万行时序数据，此处采用“单标的单行打包（JSON 字符串）”的方案。

* `symbol` (TEXT, PRIMARY KEY)：交易对唯一标识（例如：`BTCUSDT`）。
* `data` (TEXT)：打包序列化后的 JSON 字符串，结构为二维数组 `[[open_time, open, high, low, close, volume], ...]`。
* `last_updated` (INTEGER)：该标的历史数据的最后一根 K 线时间戳（毫秒）。

#### 2. 交易信号日志表 (`live_simulation_logs`)

用于结构化存储推演产生的 `live_simulation_logs` 历史流水。

* `id` (INTEGER, PRIMARY KEY AUTOINCREMENT)：自增主键。
* `time` (TEXT)：信号产生的截面时刻，格式为 `YYYY-MM-DD HH:MM:SS`。
* `action` (TEXT)：动作类型（`BUY` 或 `SELL`）。
* `coin` (TEXT)：资产代币名称（例如：`BTC`，`SOL`）。
* `direction` (TEXT)：交易方向（`LONG` 或 `SHORT`）。
* `price` (REAL)：执行时的理论资产价格。
* `reason` (TEXT)：触发该信号的策略原因（例如：`Signal Entry Long`，`大盘开关关闭`）。

### 三、 系统核心逻辑与工作流

本系统的生命周期完全围绕统一的 HTTP 请求路径进行分发与流转。

#### 1. 路由分发逻辑 (Routing Logic)

Worker 接收到 HTTP 请求后，通过解析 URL Path 执行不同的代码分支：

* `GET /`：向浏览器返回内嵌的 HTML 前端字符串，内容包括响应式布局表格与核心交互按钮。
* `GET /api/logs`：执行 SQL `SELECT * FROM live_simulation_logs ORDER BY time DESC`，将所有历史流水以 JSON 格式返回给前端。
* `POST /api/calculate`：异步触发核心拉取、计算与持久化逻辑，完成后返回状态码 `200`。

#### 2. 1m 增量数据维护逻辑 (Data Incremental Logic)

当用户点击刷新触发 `/api/calculate` 时，系统对配置的标的列表（如 BTC, ETH, SOL）执行循环：

1. **读缓存**：向 D1 表 `kline_cache` 查询指定 `symbol` 的 `last_updated` 及历史 JSON 数据。
2. **算区间**：计算 `当前系统时间 - last_updated` 之间的空缺时间窗。
3. **拉增量**：将空缺时间转换为 K 线数量（`limit`），调用币安 REST API 接口 (`/fapi/v1/klines`)，仅拉取这部分缺失的 1m K 线。如果无缓存记录，则默认全量拉取策略所需的最小根数（例如：最近 15 天约 21,600 根线）。
4. **去重合并**：在 Worker 内存中解析新老两段数组，以 `open_time` 为唯一键进行去重和升序排序。为了防止 Worker 内存溢出（OOM），保持数组最大长度不超过 30,000 根（约 20 天数据），剔除过老的数据。
5. **覆写落盘**：将合并后的新数组执行 `JSON.stringify()` 转换为字符串，通过 `UPDATE kline_cache` 覆写回 D1 数据库。

#### 3. 信号推演与指标计算逻辑 (Signal Engine Logic)

数据准备完毕后，进入纯内存计算阶段：

1. **重采样 (Resampling)**：通过高频循环遍历 1m 数组，每 240 根 1m K 线（4小时）聚合为一根 4H K 线。
* `open` = 第一根 1m 线的开盘价。
* `high` = 240 根线中的最高价最大值。
* `low` = 240 根线中的最低价最小值。
* `close` = 最后一根 1m 线的收盘价。


2. **指标矩阵化**：
* **BTC 趋势 MA**：对重采样后的 BTC 4H 收盘价数组执行滑动窗口平均计算，得出当前截面的 `btc_ma`。
* **动量 (MOM)**：计算每个标的当前收盘价相对于 $N$ 根 K 线前的价格变化率。
* **真实波幅 (ATR)**：依据公式 $\max(H-L, |H-PC|, |L-PC|)$ 计算每根 4H 线的真实波幅，并计算移动平均线得到波动率常数。


3. **状态机判定与记账**：
* 依据当前的大盘趋势（`BTC > btc_ma`），过滤并计算各代币风险调整后的动量得分。
* 在内存中虚拟重现当前的持仓状态。若检测到某个资产的持仓信号发生改变（如从无仓位变为需要多头开仓，或者原多头仓位掉出排名需要平仓），则生成一条日志对象。


4. **批量写入日志**：将新生成的日志行构建为 SQL 批量插入语句，执行 `INSERT INTO live_simulation_logs` 永久存入 D1。

### 四、 前端交互与页面逻辑

前端采用单页面应用（SPA）设计流，直接嵌入在 Worker 的主体响应中。

1. **初始化渲染**：
浏览器加载页面时，通过异步请求（JavaScript `fetch`）调用后台 `/api/logs` 接口。获取数据后，利用 DOM 操作动态在 HTML `<tbody>` 中插入行记录，展示历史的所有回测信号。
2. **状态阻断机制**：
当用户点击界面上的“实时拉取最新数据并回测”按钮时：
* **前端触发**：立即将按钮置为 `disabled` 状态，改变颜色，并将文字修改为 “⏳ 后台同步及量化推演中...”，阻断用户的二次点击。
* **后端请求**：向后台发送 `POST /api/calculate` 请求。


3. **无缝刷新流**：
* **成功分支**：当后端计算完毕返回 `HTTP 200 OK` 响应后，前端捕获该状态，自动重新触发一次 `/api/logs` 的获取流程，新产生的开平仓流水会立刻追加在表格顶部。随后解除按钮的禁用状态。
* **异常分支**：若后端返回 `HTTP 500` 错误（如币安接口限流或超时），前端捕获 `catch` 异常，并在页面顶部弹出一个红色的 Error 提示框，展示具体错误原因，同时恢复按钮的可点击状态，保护页面不发生死锁或白屏。
